# Manual Verification Runbook

Last reviewed: 2026-05-08

This runbook turns the manual-only release prerequisites into repeatable test scripts. It must be executed with real Chrome, a clean profile where noted, desensitized fixture data and the exact release zip generated by `npm run release:check`.

Do not mark any item complete without date, extension version, tester, fixture data and evidence path. Do not include API keys, Authorization headers, cookies, personal bookmarks, private URLs, page bodies or unredacted audit logs.
Use `docs/manual-evidence-templates.md` for the evidence file structure, then store completed artifacts under `release/manual-evidence/`.
Run `npm run release:manual-evidence:init` if you want the local evidence folders and files created from the current release context. The initializer does not overwrite existing files and leaves explicit placeholders that must be replaced before `npm run release:manual-evidence` can pass.

## 1. Store Screenshot Capture

Evidence path: `release/manual-evidence/screenshots/`

1. Load the release build in a clean Chrome profile.
2. Import or create desensitized fixture bookmarks for docs, tools, articles, duplicates and known broken/redirect examples.
3. Capture 6-8 screenshots:
   - Popup local search and local natural-language search.
   - Dashboard management with batch selection visible.
   - Newtab replacement with bookmark source and Speed Dial visible.
   - Health or availability workflow.
   - Privacy & Permissions Center with external request cards.
   - Backup restore preview or Recycle Bin.
   - Optional AI request preview without a real API key.
4. Record filename, date, version, viewport, fixture name, reviewer and `desensitized` status in `release/manual-evidence/screenshots/manifest.md` or equivalent text metadata.
5. Compare text in screenshots with `README.md`, `PRIVACY.md` and `docs/chrome-web-store-listing.md`.

Pass criteria: screenshots match current UI copy, use only desensitized data, show Newtab replacement honestly and do not imply AI is required.

## 2. CWS Backend Field Parity

Evidence path: `release/manual-evidence/cws-fields/`

1. Open the Chrome Web Store draft for the exact version.
2. Capture or export:
   - Short description.
   - Detailed description.
   - Category and language.
   - Permission justifications.
   - Privacy practices.
   - Support link.
   - Privacy policy link.
3. Compare each privacy-practices field with `docs/privacy-practices-mapping.md`.
4. Confirm the listing states:
   - Newtab replacement and restore path.
   - `bookmarks`, optional host permissions, `webRequest` and `webNavigation` rationale.
   - AI/Jina default off.
   - Link checks visit target URLs.
   - No remote code.
   - No sale, ads profiling, credit or lending use.
   - No default remote behavioral telemetry, without claiming no external requests.

Pass criteria: no contradiction between CWS fields, `PRIVACY.md`, listing copy, permission matrix and Options Privacy Center.

## 3. Target-User Research

Evidence path: `release/manual-evidence/user-research/`

Recruit 5-8 target users with at least 1000 bookmarks or a clear bookmark-management workflow. Record notes without personal URLs.

Tasks:

1. Install the extension and complete first search within 3 minutes.
2. Read permission/Newtab explanation and explain what data stays local and what can leave the device.
3. Find how to restore Chrome default Newtab.
4. Disable or skip AI/Jina.
5. Start a link-check preview and explain why target websites may receive requests.
6. Review an AI request preview and decide whether to proceed.

Record:

- Participant code, date and version.
- Success, hesitation and failure points for each task.
- Quotes or paraphrases with private data removed.
- P0/P1 follow-up decisions.

Pass criteria: findings are summarized, severe comprehension failures have owner and mitigation, and no private bookmark data is retained.

## 4. Visual Review

Evidence path: `release/manual-evidence/visual-review.md`

Review against accepted screenshots or baseline notes for:

- Popup.
- Options onboarding.
- Dashboard.
- Newtab.
- Privacy Center.
- Backup or Recycle Bin.

Check:

- Text does not overflow on narrow and desktop viewports.
- Primary and danger actions are visually distinct.
- Empty states include cause and next step.
- Newtab search and web-search provider controls are understandable.
- High-risk confirmations are not visually hidden or understated.

Pass criteria: visual regressions are either fixed or documented with severity, owner and release decision.

## 5. Accessibility Review

Evidence path: `release/manual-evidence/accessibility.md`

Use keyboard and a screen reader where available.

Paths:

1. Popup search, results, action menu, folder picker and modal close.
2. Dashboard search, folder filter, cards, batch selection, move/delete confirmation and return focus.
3. Newtab search, web-search engine menu, settings drawer, source picker and dashboard overlay.
4. Options navigation, Privacy Center, AI settings, backup restore preview and Recycle Bin.

Check:

- Visible focus ring.
- Logical tab order.
- Escape closes menus/modals and restores focus.
- Listbox/menu/dialog semantics are announced.
- Reduced-motion mode does not rely on animation for meaning.

Pass criteria: blocking keyboard traps, missing accessible names or unannounced destructive confirmations are fixed before submission.

## 6. High-Risk Operation Exploratory Pass

Evidence path: `release/manual-evidence/high-risk-operations.md`

Use the matrix in `docs/high-risk-operations-safety.md` as the source of rows to test.

For each applicable operation, test:

- Success.
- Cancel before mutation.
- Partial failure or stale selection.
- Missing parent folder or deleted target.
- Storage failure or backup failure where practical.
- Duplicate restore or duplicate import.
- Result summary and recovery path.

Required operation groups:

- Bookmark delete and bulk delete to Recycle Bin.
- Recycle Bin restore and clear records.
- Bookmark move and batch move.
- Folder cleanup delete, merge, move, split and split undo.
- Redirect URL update with stale URL.
- Tag import, rename, delete and clear.
- Backup restore modes.
- Audit log/history/ignore-rule clearing.
- AI apply/move suggestion paths with provider error.
- Link availability cleanup and ignore paths.

Pass criteria: no high-risk mutation runs without explicit user action, confirmation or target selection, and backup failure does not silently proceed.

## 7. Clean-Profile Platform Review

Evidence path: `release/manual-evidence/platform-review.md`

1. Create a clean Chrome profile.
2. Load the exact unpacked release zip output.
3. Open a new tab and confirm Newtab replacement disclosure.
4. Find restore instructions from Newtab and Options.
5. Use Popup search without AI.
6. Disable web search and confirm local search remains.
7. Confirm Featured Gallery is not selected by default.
8. Open Privacy Center and verify the reviewer path from `docs/permissions-matrix.md`.
9. Start link availability detection on a small fixture range and capture the optional host permission prompt for the requested target origins.
10. Deny the optional host permission once and verify the run is cancelled without visiting target sites; record the denied permission outcome.
11. Start the same fixture range again, grant the optional host permission and verify the authorized link-check path completes or records expected fixture failures.
12. Run the CWS reviewer path from `docs/chrome-web-store-listing.md`.

Pass criteria: reviewer can validate sensitive permissions in 5-10 minutes without a real AI key and without private user data.

## 8. Release Decision Record

Evidence path: `release/manual-evidence/release-decision.md`

Before CWS submission, record:

- Release zip path and sha256 from `.perf-results/release-evidence.json`.
- `npm run release:check` timestamp.
- Manual evidence paths.
- Any P0/P1 exceptions, risk, mitigation, owner and rollback plan.
- Final `submit` or `hold` decision, owner and date.

Pass criteria: no P0-A exception is unresolved, and manual evidence paths exist for every row in `docs/manual-release-evidence.md`.

After all artifacts are collected, run:

```bash
npm run release:manual-evidence
```

The command validates the expected artifact layout under `release/manual-evidence/`. It does not judge subjective screenshot quality or user comprehension; reviewers still need to sign off the notes.
